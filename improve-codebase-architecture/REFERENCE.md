# リファレンス

## 依存関係のカテゴリ

深化の候補を評価する際に、その依存関係を分類する：

### 1. In-process

純粋な計算、インメモリ状態、I/Oなし。常に深化可能 — モジュールを統合して直接テストするだけでよい。

### 2. Local-substitutable

ローカルのテスト代替物がある依存関係（例：Postgresに対するPGLite、インメモリファイルシステム）。テスト代替物が存在すれば深化可能。深化されたモジュールは、テストスイート内でローカル代替物を使ってテストされる。

### 3. Remote but owned（Ports & Adapters）

ネットワーク境界を跨ぐ自社サービス（マイクロサービス、内部API）。モジュール境界にポート（インターフェース）を定義する。深いモジュールがロジックを所有し、トランスポートは注入される。テストにはインメモリアダプターを使用する。本番環境では実際のHTTP/gRPC/キューアダプターを使用する。

推奨の形式：「共有インターフェース（ポート）を定義し、本番用のHTTPアダプターとテスト用のインメモリアダプターを実装することで、ネットワーク境界を跨いでデプロイされていても、ロジックを1つの深いモジュールとしてテストできるようにする。」

### 4. True external（Mock）

制御できないサードパーティサービス（Stripe、Twillioなど）。境界でモックする。深化されたモジュールは外部依存関係を注入されるポートとして受け取り、テストではモック実装を提供する。

## テスト戦略

核心的な原則：**重ねるのではなく、置き換える。**

- 浅いモジュールの古いユニットテストは、境界テストが存在すれば無駄になる — 削除する
- 深化されたモジュールのインターフェース境界で新しいテストを書く
- テストは内部状態ではなく、パブリックインターフェースを通じた観測可能な結果をアサートする
- テストは内部リファクタリングに耐えるべきである — 実装ではなく振る舞いを記述する

- インターフェースシグネチャ（型、メソッド、パラメータ）
- 呼び出し元がどのように使うかを示す使用例
- 内部に隠される複雑さ

## 依存関係の戦略

どのカテゴリが該当し、依存関係がどのように扱われるか：

- **In-process**: 直接統合
- **Local-substitutable**: [特定の代替物]でテスト
- **Ports & adapters**: ポート定義、本番アダプター、テストアダプター
- **Mock**: 外部サービスのモック境界

## テスト戦略

- **書くべき新しい境界テスト**: インターフェースで検証すべき振る舞いを記述する
- **削除すべき古いテスト**: 不要になる浅いモジュールのテストを列挙する
- **テスト環境の要件**: 必要なローカル代替物やアダプター

## 実装の推奨事項

現在のファイルパスに依存しない、永続的なアーキテクチャガイダンス：

- モジュールが所有すべきもの（責務）
- モジュールが隠すべきもの（実装の詳細）
- モジュールが公開すべきもの（インターフェース契約）
- 呼び出し元が新しいインターフェースに移行する方法
